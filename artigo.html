Como a internet realmente funciona: da fibra ao seu navegador

Uma viagem pelos bastidores da rede global — o que acontece no segundo entre você digitar uma URL e a página aparecer.

JB
João Bertocco 10 jun 2025 · 10 min de leitura

Você digita google.com e em menos de um segundo a página aparece. Parece mágica, mas é engenharia. Uma cadeia inteira de protocolos, servidores e roteadores trabalhou em paralelo para isso acontecer — e entender essa cadeia muda a forma como você escreve código, configura redes e diagnostica problemas.

Tudo começa com o DNS

O navegador não sabe onde fica o google.com. Ele sabe falar com endereços IP. Então a primeira coisa que acontece é uma consulta ao DNS (Domain Name System) — o sistema que traduz nomes em endereços.

Seu computador pergunta ao servidor DNS configurado no seu roteador: "qual o IP de google.com?" Se ele não souber, pergunta para um servidor acima, que pergunta para outro, até chegar nos servidores raiz que conhecem a resposta. Todo isso em milissegundos.

Curiosidade: o DNS foi criado em 1983 por Paul Mockapetris para substituir um único arquivo de texto chamado HOSTS.TXT que listava todos os computadores da internet — quando ainda cabiam em uma lista.

A conexão TCP

Com o IP em mãos, o navegador precisa estabelecer uma conexão com o servidor. Isso acontece via TCP (Transmission Control Protocol), usando o famoso three-way handshake: SYN → SYN-ACK → ACK. Três mensagens para confirmar que os dois lados estão prontos para conversar.

Por que isso é importante para um dev? Porque cada conexão TCP tem custo. É por isso que HTTP/2 reutiliza conexões e por isso que abrir muitas requisições paralelas pode, dependendo do caso, ser mais lento do que parece.

HTTP e a requisição

Com a conexão aberta, o navegador envia uma requisição HTTP GET. O servidor processa, monta a resposta (HTML, status 200) e manda de volta. Simples assim — e ao mesmo tempo, é aqui que mora boa parte da complexidade do desenvolvimento web moderno.

Renderização no navegador

O HTML chega, o navegador começa a construir o DOM. Encontra um <link> de CSS — vai buscar. Encontra um <script> — dependendo do atributo, pode bloquear tudo. É por isso que a posição dos seus scripts no HTML importa muito mais do que parece.

Entender esse fluxo completo — DNS, TCP, HTTP, renderização — é o que separa um dev que só escreve código de um dev que entende por que o código se comporta de determinada forma em produção.
